Database — PostgreSQL vs MongoDB

PostgreSQL (Neon) chosen

  • ACID transactions with SELECT FOR UPDATE — critical for atomic wallet transfers
  • Relational joins for risk scoring (txn history + CUSUM state + user in one query)
  • TimescaleDB-compatible hypertable design for time-series analytics
  • Schema enforcement prevents malformed risk snapshots
  • Neon serverless = no local DB needed, instant onboarding

MongoDB alternative

  • No native multi-document ACID before v4 — wallet double-spend risk
  • No JOINs — risk score computation needs multiple round trips
  • Schema-less = accidental omission of risk snapshot fields
  • Aggregation pipeline complex for rolling Z-score window queries
  • Better for unstructured evidence blobs (dispute files)
Verdict: The relational risk model demands SQL. CUSUM state joins + wallet row-locking + immutable R/F snapshots are all naturally expressed in PostgreSQL and awkward in MongoDB.
Frontend — React (Vite) vs Next.js

React + Vite chosen

  • Pure SPA — MetaMask injected provider works cleanly in browser context
  • Vite HMR is significantly faster than Next.js dev server for iteration
  • No SSR complexity — all data is user-specific, behind JWT auth
  • Simpler deployment — static build, any CDN
  • TanStack Query handles server-state caching without SSR hydration issues

Next.js alternative

  • SSR/SSG adds no value — pages are auth-gated, not public
  • MetaMask (window.ethereum) breaks on server-side renders
  • Heavier setup for what is fundamentally a dashboard app
  • Built-in API routes could replace Express for simpler deployments
  • Better SEO if a public landing page were needed
Verdict: MetaMask depends on window.ethereum which crashes in SSR. React+Vite is the correct choice for a wallet-integrated dashboard.
Backend — Node.js/Express vs alternatives

Node.js + Express v5 chosen

  • Same language (JS/ESM) as frontend — shared validation schemas (Zod/Joi)
  • ethers.js v6 runs natively — no FFI or subprocess for blockchain calls
  • Express v5 async error handling — no try-catch boilerplate
  • Large ecosystem: jsonwebtoken, bcrypt, emailjs all available

FastAPI (Python) alternative

  • Native numpy/scipy for statistical computation (Z-score, CUSUM)
  • Auto-generated OpenAPI docs
  • Better type safety with Pydantic
  • No native ethers.js — needs web3.py which lags behind
  • Two-language stack increases cognitive overhead
Verdict: ethers.js is the gold standard for EVM interaction. Keeping the whole stack in JavaScript eliminates the language-boundary friction for blockchain calls.
Smart contract — Solidity vs Rust (Solana)

Solidity + Hardhat chosen

  • EVM is the most mature smart contract environment
  • Hardhat local node = instant resets, console.log in contracts
  • OpenZeppelin security patterns (checks-effects-interactions)
  • Finite state machine naturally maps to Solidity modifiers
  • ABI encoding enables backend receipt verification

Rust / Solana Programs alternative

  • ~65,000 TPS vs Ethereum's ~15 TPS
  • Sub-cent transaction fees
  • No Hardhat equivalent — testing is harder
  • Account model is harder to map to escrow state machine
  • Rust learning curve much steeper than Solidity
Verdict: For a local dev/academic project the Hardhat local node is invaluable. EVM's ABI-decoded receipt verification is the foundation of the anti-spoofing architecture.
Blockchain — Local Hardhat vs Sepolia Testnet vs Mainnet

Hardhat local node chosen

  • Instant mining — no 12s block wait during development
  • Free ETH from test accounts — no faucet dependency
  • Full reset between test runs — reproducible state
  • console.log in Solidity for debugging

Sepolia testnet planned future

  • Real network conditions — latency, mempool, reorgs
  • Shareable — evaluators don't need to run a local node
  • Faucet dependency — can run dry
  • 12s block time slows iteration
Why each technology was the right call
Neon serverless PostgreSQL — why not a local DB? ▶
Neon provides a PostgreSQL endpoint accessible directly from Node.js over HTTPS. This eliminates local database provisioning entirely — evaluators clone the repo, add the DATABASE_URL to .env, and the app works. It also means the same DB works whether the backend runs on a laptop or a cloud instance. The @neondatabase/serverless driver handles connection pooling automatically, which matters because each Express request can be a separate serverless invocation.
ethers.js v6 on the backend — why use it for verification? ▶
The backend uses ethers.Interface to ABI-decode transaction receipts fetched from the Hardhat RPC endpoint. This means the backend doesn't just check that a tx hash exists — it checks that the tx emitted the correct event (EscrowCreated, EscrowReleased, etc.) with the correct parameters. A spoofing attempt would require the attacker to have the private key that signed the original transaction, making off-chain manipulation economically impractical.
TanStack Query v5 — why not Redux or Zustand? ▶
Redux and Zustand manage client-side state. The escrow platform's data is fundamentally server-driven — escrow states, wallet balances, and risk scores all live in the database. TanStack Query is purpose-built for server-state: it handles caching, background refetching, stale-while-revalidate, and loading/error states. This means the dashboard automatically reflects the latest risk score after a transaction without manual cache invalidation code.
React Hook Form + Zod — why both? ▶
React Hook Form handles form registration, submission, and error state with minimal re-renders (uncontrolled inputs). Zod provides the schema definition and type inference. The combination means the same Zod schema that validates the form also generates TypeScript types — you define the shape once and get both runtime validation and compile-time type safety. The backend uses Joi for the same schemas, so validation logic is structurally mirrored across the stack.
Checks-effects-interactions pattern in the Solidity contract ▶
Atzei et al.'s survey (cited in the report) describes the DAO hack — a reentrancy vulnerability that drained ~$60M. The pattern prevents this: (1) Check preconditions (is state valid? is caller authorised?), (2) Effect — update all state variables, (3) Interact — only then make external ETH transfers. If a malicious contract tries to re-enter during step 3, all state is already updated so the second call fails the checks in step 1.
Why simulate fiat wallet in PostgreSQL rather than using real ETH? ▶
The target users are Indian P2P participants transacting in INR. Requiring them to acquire and manage ETH creates a UX barrier and regulatory complexity. The simulated wallet lets the platform demonstrate escrow mechanics — lock, release, dispute, refund — using rupee-denominated amounts, while the Ethereum smart contract handles the trustless enforcement layer using a corresponding ETH representation. This hybrid mirrors how real fintech products (like Razorpay Escrow) work under the hood.
Likely interview / viva questions
hard Why PostgreSQL and not a time-series DB like InfluxDB for the transaction log? ▶
InfluxDB is optimised for append-only metrics with no relational context. The risk model needs to join the transaction log with the user profile, CUSUM state, and escrow record in a single query. InfluxDB has no concept of foreign keys or joins. PostgreSQL with a hypertable design (TimescaleDB-style partitioning on created_at) gives time-series query performance while retaining full relational capability. If we needed sub-millisecond IoT-scale writes, InfluxDB would win — but escrow transactions are low-frequency and high-relational.
hard What happens if the Hardhat node goes down after a tx is submitted but before the backend verifies it? ▶
The backend receipt verification uses ethers.JsonRpcProvider.getTransactionReceipt(txHash). If the node is down, this throws — the backend returns a 503 and the database write never happens. The escrow stays in its previous state. The user must retry. This is correct behaviour — it's better to require a retry than to accept an unverified state update. In a production system this would be handled by a message queue (BullMQ, SQS) that retries the verification job with exponential backoff.
hard CUSUM requires a stable mean and sigma. What happens for a new user with 2–3 transactions? ▶
This is the cold-start problem. With fewer than ~5 transactions, sigma is either zero or extremely small, making the CUSUM sensitivity very high — almost any transaction would trigger an alarm. The system initialises sigma to amount × 0.3 as a fallback when history is thin. The correct production solution is a cold-start phase (already in the schema as is_restricted_phase) where new users have fixed, conservative escrow limits regardless of risk score until they accumulate enough history for the statistical detectors to calibrate properly. This is noted in the report as designed but not yet implemented.
medium Why is the Gompertz function used for lock duration rather than a simpler linear formula? ▶
A linear formula like T = T_min + (1 - R) × range produces abrupt changes near the boundaries. The Gompertz function is an S-curve — it changes slowly at extreme values of R and F (very trusted or very risky users) and steeply in the middle transition zone. This means a slightly anomalous transaction for a highly trusted user doesn't dramatically increase their lock time, and a slightly improved score for a high-risk user doesn't dramatically reduce it. The non-linearity creates hysteresis that makes the system harder to game through small, incremental trust-building followed by a large fraud attempt.
medium Why JWT over session cookies for authentication? ▶
JWT is stateless — the server doesn't need to look up a session in the database on every request. The token is verified by the middleware using the secret key alone. This matters for the escrow platform because multiple escrow state updates, wallet queries, and risk score fetches all happen in rapid succession during a user session. Stateless auth means each of these requests is independently verifiable without a session-store round trip. The downside is that JWTs cannot be invalidated before expiry — a logout just deletes the token client-side. For high-security operations (wallet withdrawals), the 6-digit security PIN provides a second factor that compensates for this limitation.
medium React vs Next.js — when would you choose Next.js for this project? ▶
If the escrow platform needed a public-facing marketing page, transaction explorer, or dispute tracker that search engines should index, Next.js SSR/SSG would be justified. Also, if MetaMask were replaced by a custodial wallet (where private keys live server-side), the SSR constraint disappears and Next.js API routes could eliminate the separate Express server, simplifying deployment. For the current architecture — private dashboard + MetaMask + separate Express backend — React+Vite is strictly better.
conceptual What is the Byzantine Generals problem and does it apply here? ▶
The Byzantine Generals problem asks how distributed nodes can reach consensus when some nodes may be malicious or faulty. Ethereum's consensus mechanism (Proof of Stake post-Merge) solves this by requiring validators to stake ETH as collateral — dishonest validators lose their stake (slashing). This project uses a local Hardhat node which has a single validator (the node itself), so Byzantine fault tolerance is irrelevant locally. On Sepolia testnet it becomes relevant — the system relies on Ethereum's consensus finality for the anti-spoofing guarantee. The 6-block confirmation wait recommended for production ETH transactions is the practical implication.
